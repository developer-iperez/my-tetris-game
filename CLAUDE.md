## Project status

Initial scaffold implemented in TypeScript (OOP). `README.md` remains a captured planning conversation (in Spanish) about the intended architecture; the sections below describe what actually exists now.

## Build & run

- `npm install` — install dev dependencies (TypeScript, browser-sync, concurrently; no runtime deps).
- `npm run build` — compile `src/**/*.ts` to `dist/` via `tsc` (strict mode).
- `npm run watch` — incremental compile on change.
- `npm run serve` — build, then serve the repo root on port 8080 via `browser-sync`, which live-reloads the browser whenever `dist/`, `css/`, `index.html`, or `manifest.json` change.
- `npm run dev` — build once, then run `watch` and `serve` together (equivalent to the VS Code "Debug Tetris (Chrome)" launch config's preLaunchTask).
- The VS Code debugger (`.vscode/launch.json` → "Debug Tetris (Chrome)") runs a blocking `tsc` build first, then starts `tsc --watch` + `browser-sync` in the background, so the debug session always starts from freshly compiled output and live-reloads on save.
- `service-worker.js` caches `dist/*.js` cache-first with no revalidation; `src/app.ts` skips registering it (and clears any previously registered one) on `localhost`/`127.0.0.1` so local iteration never serves a stale cached build. It still registers normally in production.

There is no bundler: `dist/` mirrors `src/`'s structure and is loaded directly via ES module `<script type="module">` tags. `dist/` is generated and gitignored — run `npm run build` before opening `index.html` or deploying.

## Architecture

Browser-based Tetris, static frontend only, deployable to GitHub Pages (no server-side code):

- **Language**: TypeScript, compiled with `tsc` (`strict: true`), OOP design.
- **Rendering**: Canvas 2D, no rendering library.
- **Game engine decoupled from UI**: `src/engine/` (`Board`, `Piece`, `Game` classes) has no dependency on rendering or input, so the frontend could later be swapped for a framework without rewriting the engine.
- **Installable as a PWA**: `service-worker.js` + `manifest.json` provide offline support and installability. The service worker cache list must be kept in sync with `dist/` output paths.
- **Directory layout**:
  ```
  /
  ├── index.html
  ├── css/
  ├── src/
  │   ├── engine/       # Board, Piece, Game — pure game logic, no DOM/canvas deps
  │   ├── renderer/      # CanvasRenderer — reads Game state, draws to <canvas>
  │   ├── ui/            # Hud, KeyboardControls
  │   ├── network/       # multiplayer/WebRTC signaling client (not yet implemented)
  │   └── app.ts         # TetrisApp: wires engine + renderer + ui, owns the game loop
  ├── dist/              # compiled JS output (gitignored, generated by `npm run build`)
  ├── assets/
  ├── manifest.json
  └── service-worker.js
  ```
- **Multiplayer (not yet implemented)**: planned via WebRTC for peer-to-peer gameplay data, since GitHub Pages cannot run server-side code. WebRTC still requires a small external signaling server (e.g. a lightweight Node.js WebSocket service) purely to exchange initial connection info (SDP/ICE) — that signaling server is not part of this repo's deployment and does not participate in actual gameplay. Same-network-only and Bluetooth-based multiplayer were considered and rejected (browser sandboxing prevents raw sockets; Web Bluetooth has poor cross-browser support, especially Safari).

## Key classes

- `Board` (`src/engine/board.ts`): owns the grid, collision/bounds checks, merging a piece in, line clearing.
- `Piece` (`src/engine/piece.ts`): a tetromino's shape/position; `rotatedCells()` is pure (doesn't mutate), `clone()` is used to trial moves before committing.
- `Game` (`src/engine/game.ts`): orchestrates `Board` + `Piece` — movement, rotation with wall-kicks, gravity `tick()`, locking, scoring/leveling. No DOM or timing code.
- `CanvasRenderer` (`src/renderer/canvas-renderer.ts`): reads a `Game` and draws it; holds no game state of its own.
- `TetrisApp` (`src/app.ts`): the only class that owns a `setTimeout`-based game loop; wires everything together at startup.
